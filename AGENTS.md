# AGENTS.md

## Project Purpose

`senna-automation` is the public website and content hub for Senna Automation.
It markets AI workflow automation, custom software, and operations consulting,
and it is the source for blog content that feeds downstream social-post
automation.

Production website:

```text
https://www.senna-automation.com
```

This repo owns the public Next.js site. It does not own the VPS infrastructure,
n8n workflows, Mixpost, Umami, or the Remotion renderer, but the business
depends on those systems and this site integrates with several of them.

## System Map

- Website repo: `~/Projects/senna-automation`
- Infra repo: `~/Projects/infra`
- Video renderer repo: `~/Projects/senna-overview-video`
- Live VPS SSH host: `senna-infra`
- Public website host: Vercel for `senna-automation.com` and
  `www.senna-automation.com`
- Infrastructure host: VPS subdomains such as `n8n.senna-automation.com`,
  `mixpost.senna-automation.com`, and `umami.senna-automation.com`

Keep the apex and `www` website records pointed at Vercel. Infra subdomains are
routed to the VPS through the separate `infra` project.

## Website Stack

- Next.js App Router, React, TypeScript.
- MUI is the primary component system.
- Framer Motion is used for motion-heavy interface pieces.
- Blog content is Markdown under `src/content/blog`.
- Feeds are generated by route handlers for RSS, Atom, and JSON Feed.
- Supabase/Postgres supports the live chat message/session flow.
- Telegram is used as the operator-side chat destination.
- Umami analytics loads in production from `umami.senna-automation.com`.
- Contact form submissions go through `/api/form-hook`, which rewrites to a
  Make webhook in `next.config.ts`.

Important files:

- `src/app/page.tsx`: home page.
- `src/app/about/page.tsx`, `src/app/services/page.tsx`,
  `src/app/solutions/page.tsx`, `src/app/pricing/page.tsx`,
  `src/app/contact/page.tsx`: main marketing pages.
- `src/app/blog/page.tsx` and `src/app/blog/[slug]/page.tsx`: blog listing and
  detail pages.
- `src/app/layout.tsx`: global metadata, JSON-LD, app chrome, analytics, chat,
  and privacy notice.
- `src/app/api/chat/*`: chat session, send, and poll endpoints.
- `src/app/api/telegram/webhook/route.ts`: Telegram inbound webhook.
- `src/utils/site.ts`: canonical site URLs and feed constants.
- `src/utils/db.ts`: Postgres pool for server routes.
- `src/utils/supabase/*`: Supabase clients.
- `src/utils/telegram.ts`: Telegram Bot API helpers.
- `src/app/tokens.css`, `src/components/theme/*`, and `specs/`: design tokens
  and component guidance.
- `trackedEvents.md`: analytics event map.
- `docs/blogToMixpost.md`: older implementation notes for blog-to-social
  automation.

## Local Development

Install and run the site:

```bash
cd ~/Projects/senna-automation
npm install
npm run dev
```

Open:

```text
http://localhost:3000
```

Useful checks:

```bash
npm run lint
npm run build
npm run token-audit
```

`next.config.ts` currently allows production builds to complete despite
TypeScript or ESLint errors. Do not treat a successful build as proof that the
change is clean; run lint and inspect warnings separately.

## Editing Guidance

Follow existing page, component, and token patterns. Prefer established MUI
theme usage and the tokens in `src/app/tokens.css` over hardcoded visual values.
When changing CSS-heavy surfaces, run:

```bash
npm run token-audit
```

For page changes, check desktop and mobile layouts. The site is a polished
business-facing marketing surface; avoid generic landing-page filler, unrelated
decorative elements, and copy that over-explains the UI.

For blog posts:

- Add Markdown in `src/content/blog/<slug>.md`.
- Add images under `public/blog/<slug>/` or `public/blog/` as appropriate.
- Keep slugs stable once published.
- Validate the blog detail page, feed routes, and Open Graph image route.
- After publishing, use IndexNow and WebSub when relevant:

```bash
npm run indexnow:submit -- /blog/<slug>
npm run websub:publish
```

## Business Infrastructure

The `infra` repo is the operational backbone for the business. It runs the VPS
Docker Compose stack and should be treated as a separate production system.

Local path:

```text
~/Projects/infra
```

Server path:

```text
senna-infra:~/Projects/infra
```

Core services include:

- Traefik for HTTPS routing and certificates.
- n8n and n8n workers for automation workflows.
- Postgres and Redis for shared service persistence/queueing.
- Mixpost Pro for social media drafts and scheduling.
- Umami for website analytics.
- Mautic and Akaunting for business operations.
- OpenVPN/OpenClaw/ngrok/CoreDNS support services.
- The live server compose also includes `remotion-render`, built from
  `~/Projects/senna-overview-video`, for Instagram video generation.

Do not make infra changes in this repo. If a website change requires infra
support, update and deploy `~/Projects/infra` separately.

Useful server checks:

```bash
ssh senna-infra 'cd ~/Projects/infra && docker compose ps'
ssh senna-infra 'cd ~/Projects/infra && docker compose logs --tail=100 n8n'
ssh senna-infra 'cd ~/Projects/infra && docker compose logs --tail=100 traefik'
```

## Automation And Social Posting

Blog posts on this website are used by n8n workflows to create social content.
The modern flow is more capable than the older `docs/blogToMixpost.md` notes:

- n8n can generate posts for all channels from a new blog post.
- n8n can generate an individual channel post in isolation.
- Instagram video generation is async: n8n submits a Remotion render job,
  returns quickly, and a scheduled completion workflow later creates the Mixpost
  draft after the MP4 is ready.
- Mixpost stores social drafts/scheduled posts.
- The Remotion renderer lives in `~/Projects/senna-overview-video`.

When changing blog metadata, feeds, URLs, or Open Graph behavior, consider the
impact on n8n social workflows and Mixpost drafts.

## Remotion Renderer Relationship

The Instagram video system is intentionally outside this website repo:

```text
~/Projects/senna-overview-video
```

That repo owns Remotion compositions, renderer API endpoints, async render job
storage, sample props, visual QA, and deployment of the `remotion-render`
container.

Do not edit Remotion video code here. Make renderer changes in
`senna-overview-video`, then deploy from that repo.

## Deployment

Website deployment is separate from VPS infra deployment. Normal website changes
should be committed and pushed through the repository flow used by Vercel.

Before handing off a website change:

```bash
npm run lint
npm run build
git status --short
```

For SEO/content releases, also consider:

```bash
npm run indexnow:submit -- --all --dry-run
npm run websub:publish
```

Only deploy or restart VPS services from `~/Projects/infra` when the change
requires it. Only deploy the Remotion renderer from `~/Projects/senna-overview-video`.

## Boundaries

- Keep this repo focused on the public website, content, website APIs, and
  website-adjacent scripts.
- Do not place runtime data, backups, or service secrets in this repo.
- Do not edit live n8n workflows without explicitly working in n8n.
- Do not change Mixpost, Traefik, Docker Compose, or database service config
  from this repo.
- If a website change affects automation inputs, document or test the related
  n8n/Mixpost behavior before considering the change complete.
